# Notas de sesion - 2026-02-07

## Objetivo

Evaluar y estabilizar la capa de tiempo real en ESP32 para publicar estado de tablero con Microdot.

## Decisiones tomadas

- Se descarto WebSocket como canal principal para este caso.
- Se adopto SSE para actualizaciones servidor -> cliente.
- Se mantuvo `POST /move` para entrada de movimientos desde UI.
- Se agrego `GET /api/history` para recuperacion de eventos faltantes.

## Motivos del cambio (WebSocket -> SSE)

- El caso de uso es unidireccional: el servidor publica estado.
- SSE simplifica cliente web con `EventSource` y reconexion automatica.
- Menor complejidad operativa para ESP32 frente a manejo de multiples sockets WS.
- Menor codigo de aplicacion para pruebas y mantenimiento.

## Implementacion

- Se creo modulo propio `modules/sse/SSE.py`.
- El stream SSE conserva solo el ultimo mensaje en buffer (`_buffer`).
- `main.py` publica estado minimo: `move` y `mem_free`.
- Se agrego historial acotado en memoria para sincronizacion.

## Riesgos y mitigaciones

- Riesgo: al conservar solo ultimo evento, se pueden pisar updates intermedios.
- Mitigacion: ids incrementales y fallback a `GET /api/history` ante huecos.
- Riesgo: consumo de RAM con demasiados clientes o payload grande.
- Mitigacion: payload minimo y limite de historial.

## Estado actual

- SSE funcionando en ESP32 con ruta `/events`.
- Pagina de prueba operativa en `/`.
- Endpoints activos: `/move`, `/api/state`, `/api/history`.
